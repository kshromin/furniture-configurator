# Синхронизация между компьютерами — что делать

Раньше код между двумя компьютерами расходился через Yandex.Disk (копировались все файлы, включая `.git`). Это ненадёжно: `.git` — не обычные файлы, а база данных git, файловая синхронизация может её испортить при не вовремя пойманном моменте. Поэтому теперь код синхронизируется через GitHub, а Yandex.Disk синхронизирует только сами файлы проекта (это происходит само собой, ничего для этого делать не нужно).

Репозиторий: **https://github.com/kshromin/furniture-configurator** (приватный).

## Если садишься за второй компьютер первый раз после этого перехода

Проверь на этом компьютере, подключён ли уже локальный git-репозиторий к GitHub:

```
cd путь\до\Config
git remote -v
```

**Если `git remote -v` ничего не показал** (или ругается «not a git repository» — тогда сначала `cd` в правильную папку) — значит этот компьютер ещё не знает про GitHub. Подключи и подтяни актуальный код:

```
git remote add origin https://github.com/kshromin/furniture-configurator.git
git fetch origin
git branch -M main
git reset --hard origin/main
```

⚠️ `git reset --hard` **сотрёт** любые локальные незакоммиченные изменения на этом компьютере и заменит историю той, что на GitHub. Перед этим стоит проверить `git status` и `git log --oneline -5` — если там есть что-то новое, чего не было на другом компьютере (например, кто-то тут уже работал с Claude до перехода на GitHub), **сначала разберись, что это, и сохрани при необходимости** (спроси Claude — покажи ему `git status`/`git diff`, прежде чем стирать).

При первом `git push` или `git pull` через HTTPS откроется окно входа в GitHub в браузере — войди под аккаунтом `kshromin`, дальше вход будет запоминаться.

## Обычный рабочий цикл (на любом компьютере, каждый раз)

**Перед началом работы:**
```
git pull
```

**После того как поработали и закоммитили:**
```
git push
```

Если открываешь эту папку с Claude — можно попросить его сделать `git pull` в начале сессии и `git push` в конце (коммитить он сам не станет без явной просьбы — так настроено намеренно).

## Если git внезапно «не видит» ни одного коммита (все файлы untracked)

Случилось 21.07.2026 при смене компьютера: YandexDisk синхронизирует папку целиком, включая
`.git`, и при переносе состояния с другого компьютера потерял файл `.git/refs/heads/main` и
испортил индекс — `git status` показывал ВСЕ файлы проекта как неотслеживаемые, `git log` — «ветка
main не имеет коммитов». Сами данные (объекты, история) при этом целы.

Починка (без потери файлов, ничего не удалять и не клонировать заново):

1. `tail .git/logs/HEAD` — последняя строка, второй хеш = последний коммит (вершина ветки).
2. `git update-ref refs/heads/main <этот-хеш>` — вернуть ветку на место.
3. `git reset` (БЕЗ `--hard`!) — перестроить индекс от восстановленного HEAD, рабочие файлы не
   трогаются.
4. `git fsck --no-dangling` — убедиться, что база цела; `git status` теперь показывает только
   реальные изменения.

После этого стоит проверить `git diff`: рабочие файлы могли приехать с другого компьютера
**новее** восстановленного коммита (незакоммиченная там работа) — это нормальные правки, их
надо разобрать и закоммитить, а не откатывать.
