## 2026-07-12/13 — сессия 14: перетаскивание наполнения мышкой, верхняя полка стала обычным элементом

Общие на тип параметры (высота ящика, глубина сетки, цвет корзины и т.д.) остались групповыми полями секции — индивидуальна только позиция.
- **Мышью можно перетаскивать вверх-вниз любой элемент секции, кроме торцевого вешала** (`js/core/itemDrag.js`, новый модуль): raycast по мешам `furnitureGroup`, драг ограничен плоскостью, повёрнутой к камере, через мировую позицию элемента. Во время движения элемент свободно проходит сквозь другие (двигаем меши напрямую, без проверки на каждый кадр); при пересечении с другим элементом или выходом за `fillBottom/fillTop` — подсветка красным (`emissive`); если отпустить мышку в момент подсветки — позиция не коммитится, `buildFurniture()` возвращает геометрию на место. Торцевое вешало — отдельная ветка: не двигается свободно, а прыгает («снап») к ближайшей полке-кандидату при отпускании, крепится через `sec.valetAnchorId`.

Удалить её через UI нельзя (нет крестика-чипа + защита в обработчике), но по высоте она полностью свободна — раньше зона над ней была искусственно недоступна («здесь нельзя разместить элемент»), теперь весь диапазон открыт для любых элементов.
- **`sectionVerticalBounds()`**: убран отдельный `topShelfY`, остались только `fillBottom`/`fillTop` — единый валидный диапазон для абсолютно любого элемента секции, включая саму верхнюю полку.
- **Добавление нового элемента** (кнопка «+» на вкладке «Внутр.») ищет первое свободное место по высоте (`findFreeSlot`), если места нет — тост «Нет места для нового элемента в этой секции»; штанга по умолчанию предпочитает место сразу под верхней полкой (как раньше), а не первое свободное снизу.
- **UI** (`tabs.js`): счётчик по типу заменён на количество экземпляров в `items` + чип с крестиком на каждый (кроме `pinned`); ширина боковой панели увеличена 300→340px (`css/style.css`, `#panel`/`#viewport`/`#sidebar`) — чекбокс «без доводчика» и выбор цвета корзин перестали перетекать на вторую строку.

## 2026-07-17 — сессия 28: фиксация просветов между элементами секции

**Что сделано:**
- **Фиксация размеров** (задание «фиксация размеров 17,07»): у каждой статичной размерной
  подписи (просвет между соседними элементами секции, `js/core/dimensions.js`) появилась
  галочка — фиксирует именно этот просвет. Хранится в `sec.lockedGaps` (массив ключей —
  id элемента, ограничивающего просвет сверху, либо `'ceiling'` для самого верхнего; тот же
  принцип ключей, что у `backWallSegments`/`'floor'`), переживает сохранение/загрузку
  прорисовки. Уточнял с пользователем два момента: (1) когда зафиксированный просвет должен
  был бы измениться — двигается следующий СВОБОДНЫЙ элемент по цепочке, а не блокировка/ошибка;
  (2) фиксация — часть state секции, не временная UI-подсказка.
- **Алгоритм каскада** (`resolveLockedMove`, `sectionGapsKeyed` в `wardrobe-items.js`): при
  сдвиге элемента цепочка расширяется в обе стороны, пока соседний просвет зафиксирован;
  первый незафиксированный просвет поглощает сдвиг (подрезается по месту упора в соседа/
  пол/потолок). Отдельный краевой случай, найденный и исправленный в процессе тестирования:
  если цепочка упирается в САМ пол/потолок секции, а прилегающий к нему просвет тоже
  зафиксирован — двигать уже некого, и по вычислению сдвиг должен быть НЕ «подрезан до упора»
  (как со свободной границей), а запрещён целиком в любую сторону (сохранение фиксированного
  расстояния до неподвижного пола/потолка иначе невозможно). Первая версия ошибочно подрезала
  вместо полной блокировки — поймано и исправлено серией прямых тестов через `preview_eval`
  (обычный сдвиг / каскад через 1 и 2 элемента / упор в границу с подрезкой / полная блокировка
  при зафиксированном упоре).
- **Подключено в двух местах** (`js/core/itemDrag.js`): ввод точного числа в поле просвета
  (`commitGapEdit`) и обычное перетаскивание мышкой (`onPointerMove`/`onPointerUp`) — раньше
  обе ветки напрямую двигали только сам элемент и блокировали пересечения тостом; теперь обе
  зовут `resolveLockedMove` и применяют позиции всей цепочки. При живом драге меш каскадного
  соседа не следует за мышью в реальном времени (едет только сам двигаемый элемент) — сосед
  довернётся в верную позицию одним кадром на отпускании (`buildFurniture()` всё равно
  перерисовывает всё разом) — сознательное упрощение объёма работ, при желании можно доделать
  живое сопровождение соседей.
- Галочка — единственный интерактивный элемент в `.dim-label` (у оверлея `pointer-events: none`
  ради вращения камеры мышкой) — `pointer-events: auto` только на самой галочке
  (`css/style.css`), тот же паттерн, что чинил в сессии 25 для кнопки толщины полки.

**Почему:** пользователь заметил, что при вводе желаемого размера просвета остальные размеры
меняются непредсказуемо — предложил фиксацию конкретных просветов галочкой.

**На заметку:**
- Живое 3D-перетаскивание мышкой не перепроверено вручную — тестовый аккаунт `test@test.com`
  удалён (см. сессию 25, «Тестовые аккаунты удалены»), без логина канвас не рендерится
  (`#viewport` держится на `display:none` до успешной авторизации). Логика идентична уже
  проверенному пути ввода точного числа — тот же `resolveLockedMove` — но стоит перепроверить
  мышкой на реальном аккаунте.
- Пользователь подтвердил «вроде все норм» без ручной проверки мышкой — если после реальной
  проверки найдутся шероховатости именно в live-драге (а не в самой логике фиксации), скорее
  всего дело в упомянутом выше упрощении (соседи не едут в реальном времени).
